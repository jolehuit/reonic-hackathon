Creating Your First Dataset
Ready to get started?

Upload a JSON, JSONL, or CSV file (up to 50 MB / 500,000 rows) to create a dataset and start fine-tuning.

Upload a Dataset
Shell

# List your datasets
curl -X GET https://api.pioneer.ai/felix/datasets \
  -H "X-API-Key: REDACTED"
Pioneer generates data for any task. Open a project from Research to create or browse datasets.

Supported File Formats
Pioneer accepts JSON, JSONL, and CSV files up to 50 MB. JSONL (one JSON object per line) is recommended for large datasets.

Dataset Types
Pioneer supports four dataset types, each with a specific schema. Choose the type that matches your fine-tuning task when uploading.

Named Entity Recognition (NER)
Train a model to extract named entities (people, organizations, locations, products, etc.) from text. Each row needs a text field and an entities array of [span, label] pairs.

jsonc

// Each entity span must appear in the text (case-insensitive)
{
  "text": "Apple launched the iPhone in San Francisco.",
  "entities": [
    ["Apple", "ORG"],
    ["iPhone", "PRODUCT"],
    ["San Francisco", "LOC"]
  ]
}
Entity spans must be exact substrings of the text field.
Use consistent entity type names across all rows (e.g. always ORG, not sometimes organization).
A row may have zero entities, but the entities array must be present.
Text Classification
Train a model to assign one or more labels to a piece of text. Each row needs a text field and either a single label string or a labels array.

jsonc

// Single-label classification
{
  "text": "Great product, highly recommend!",
  "label": "positive"
}

// Multi-label classification
{
  "text": "Apple announces new AI chip for data centers",
  "labels": ["technology", "business"]
}
Use label (string) for single-label or labels (array of strings) for multi-label -- do not mix both in one dataset.
Labels should be consistent strings (e.g. "positive", "negative", "neutral").
JSON Extraction
Train a model to extract structured fields from unstructured text (prices, dates, product names, etc.). Uses the same schema as NER -- text plus entities as [span, label] pairs.

jsonc

{
  "text": "The MacBook Pro costs $2,499 at the Apple Store in NYC.",
  "entities": [
    ["MacBook Pro", "PRODUCT"],
    ["$2,499", "PRICE"],
    ["Apple Store", "RETAILER"],
    ["NYC", "LOCATION"]
  ]
}
Decoder (Chat SFT)
Fine-tune a decoder (causal language) model on conversational data. Each row must contain a messages array with objects that have role and content fields.

jsonc

// JSONL format recommended (one JSON object per line)
{"messages": [
  {"role": "system", "content": "You are a helpful coding assistant."},
  {"role": "user", "content": "Write a Python function to sort a list."},
  {"role": "assistant", "content": "def sort_list(items):\n    return sorted(items)"}
]}
Allowed roles: system (optional), user, and assistant.
The system message is optional and should appear first if present.
JSONL format is recommended for large datasets.
Common External Formats
Pioneer automatically detects and converts these popular formats when you upload:

Format    Fields    Conversion
OpenAI / ChatML    messages    Native format -- no conversion needed
Alpaca    instruction, input, output    instruction + input = user, output = assistant
ShareGPT    conversations: [{from, value}]    human = user, gpt = assistant
Prompt / Output    prompt, output    prompt = user, output = assistant
Instruction / Response    instruction, response    instruction = user, response = assistant
Create via API (Upload)
Shell

# Step 1: Get a presigned S3 upload URL
UPLOAD=$(curl -s -X POST https://api.pioneer.ai/felix/datasets/upload/url \
  -H "X-API-Key: REDACTED" \
  -H "Content-Type: application/json" \
  -d '{"dataset_name":"YOUR_DATASET_NAME","dataset_type":"classification","format":"csv"}')
PRESIGNED_URL=$(echo $UPLOAD | jq -r '.presigned_url')
DATASET_ID=$(echo $UPLOAD | jq -r '.dataset_id')

# Step 2: Upload file directly to S3 (no size limit)
curl -X PUT "$PRESIGNED_URL" \
  -H "Content-Type: application/octet-stream" \
  --data-binary @./dataset.csv

# Step 3: Trigger processing
curl -X POST https://api.pioneer.ai/felix/datasets/upload/process \
  -H "X-API-Key: REDACTED" \
  -H "Content-Type: application/json" \
  -d "{\"dataset_id\":\"$DATASET_ID\"}"

# Wait for status=ready before training
curl -X GET https://api.pioneer.ai/felix/datasets/YOUR_DATASET_NAME \
  -H "X-API-Key: REDACTED"
Valid dataset_type values: ner, classification, custom, decoder. Training requests fail if the dataset does not exist or is still processing.

Option 1: Upload Your Own Data
If you already have labeled data, upload it from the dataset creation page directly. Choose your dataset type, map columns if needed, and Pioneer validates your data before saving.

Option 2: Import from HuggingFace
Search and import any compatible dataset from the HuggingFace Hub. Pioneer will automatically convert it to the correct format. Try it from the dataset creation flow inside a Research project.

Option 3: Synthetic Data Generation
Start from zero data. Pioneer uses Felix, a proprietary data-generation engine, to create high-fidelity synthetic training datasets. Describe the entities or classes you care about, provide a few optional seed examples, and expand into thousands of diverse samples.

Define your entity types or classification labels.
Optionally provide a handful of seed examples to guide the style and domain.
Choose the number of samples to generate.
Felix generates diverse, realistic samples with accurate annotations.
The fastest way to generate data is through the Agent. Describe what you need in natural language and the agent will configure and run the generation job for you.

Text

Agent prompt example:
"Generate 500 NER samples for extracting product names,
prices, and store locations from retail descriptions."
Best Practices
Preview, edit, and delete individual samples from the dataset detail view.
Use the built-in duplicate remover to clean noisy data.
Review generated samples before adding them to your dataset. Synthetic data can drift from your real-world distribution.
Be explicit about domain context: specify your industry, sector, and the setting your samples should reflect.

Terminology
Key concepts you'll encounter throughout Pioneer.

Small Language Models (SLMs)
SLMs are models trained to be highly accurate on specific tasks rather than generally capable across all tasks. Think of them as specialists versus generalists — a 205M parameter model trained on your data will outperform many LLMs on your specific use case, at a fraction of the cost and latency. Smaller doesn't mean weaker — it means precise.

Datasets
A dataset is your labeled examples — the input/output pairs that capture what "correct" looks like for your task. Everything downstream (training, evals) is only as good as this. Pioneer expects them in JSONL format: one example per line, each with an input and the expected output.

Training
In Pioneer's context, training means fine-tuning — you're not building a model from scratch, you're taking a pre-trained base model and continuing to update its weights on your dataset until it gets good at your specific task. Think of the base model as knowing language; fine-tuning teaches it your domain. Pioneer uses parameter-efficient methods under the hood so you don't need millions of examples to see real gains.

Evaluations
Evals are how you know whether any of this worked. You take a held-out set of examples the model has never seen, run inference, and score the outputs against what you expected. Without evals, you're flying blind — a model can look great in demos and quietly fail on real data. Think of evals as your test suite: run them before you ship anything.

Inference
You've built a model. Inference is just using it. You pass in an input, the model runs a forward pass through its weights, and you get an output back. That's it — no learning, no weight updates, just computation. In Pioneer's case, that means hitting an API endpoint and getting structured predictions back in milliseconds.

Continuous Adaptation
This is not real-time learning. Continuous adaptation is a structured cycle: production logs accumulate, Pioneer's agents curate a training dataset, a fine-tuning job runs, and the best checkpoint is evaluated before anything touches production. You control when a new model version gets promoted — nothing updates automatically without clearing your defined quality threshold.

API Reference
The Pioneer API lets you run inference, manage datasets, start training jobs, and evaluate models programmatically. Pioneer also exposes OpenAI- and Anthropic-compatible endpoints so you can integrate with any tool that supports those formats.

Base URL: https://api.pioneer.ai

Recommended API Order
Create or upload a dataset, then confirm it is ready.
Start training with that exact dataset name.
Poll training status until it is complete.
Run an evaluation using the completed training job ID and the dataset name.
Run inference with model_id set to your training job ID (or a base model ID like fastino/gliner2-base-v1).
Replace placeholders (for example YOUR_DATASET_NAME and YOUR_TRAINING_JOB_ID) before running commands.

Rate Limits
The global default is enforced per client IP. Per-endpoint limits are enforced per authenticated user and stack on top of the global default. Requests that exceed a limit receive 429 Too Many Requests with a Retry-After header.

POST /gliner-2/*    Per user    15,000 / min
